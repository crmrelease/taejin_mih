# 좋아요 축구선수 — 입력&조회 보드 기획서

> 작성: 솜이 (기획자)
> 작성일: 2026-05-09
> 수신: 랑이 (개발자/메이커), 모 (모니터링)
> 상태: 개발 착수 요청

## 1. 한 줄 요약

이름과 좋아하는 축구선수를 적으면, 다른 사람들이 적어둔 목록도 같이 볼 수 있는 **한 페이지짜리 미니 게시판**을 만들어주세요. GitHub Pages 같은 상용 링크로 누구나 바로 접속해서 쓸 수 있게요.

## 2. 왜 만드나요 (배경)

- 두 번째 미니 제품. 계산기보다 한 단계 발전 — **데이터를 저장하고 다시 불러오는 흐름**까지 포함시켜요.
- 데이터 입력 이벤트가 발생하니까, 모가 **입력 모니터링/알림 환경 구축**을 실험해볼 수 있어요. (모니터링 환경은 모 영역, 이 기획서에는 "외부에서 관찰 가능해야 한다"는 요구만 명시)

## 3. 사용자 시나리오

1. 사용자가 상용 링크로 페이지를 엽니다.
2. 상단에 **입력 영역**, 하단에 **조회 영역**이 보여요.
3. 입력 영역에 `이름`, `좋아하는 축구선수`를 적고 **등록** 버튼을 누릅니다.
4. 등록 즉시 하단 조회 영역 맨 위에 방금 입력한 항목이 추가돼요.
5. 페이지를 새로고침해도, 다른 기기/브라우저에서 열어도 **지금까지 입력된 모든 항목**이 보입니다.

## 4. 꼭 들어가야 하는 기능 (Must)

| 항목 | 설명 |
|---|---|
| 입력 폼 | `이름` (텍스트), `좋아하는 축구선수` (텍스트) 두 필드 + **등록** 버튼 |
| 입력 검증 | 둘 다 비워두면 등록 안 되고 안내 문구 표시. 길이는 각 30자 이내 |
| 조회 목록 | 지금까지 등록된 모든 항목을 카드/리스트로 표시. **최신이 위** |
| 영구 저장 | 새로고침/다른 기기에서 열어도 동일한 목록이 보여야 해요 (브라우저 localStorage 만으로는 부족) |
| 즉시 반영 | 등록 버튼 누른 직후 목록에 추가됨 (수동 새로고침 없이) |
| 빈 상태 안내 | 등록된 항목이 0개일 때는 "아직 아무도 등록 안 했어요" 같은 안내 문구 |

## 5. 있으면 좋은 기능 (Nice to have)

- 등록 시각 표시 (`방금 전`, `3분 전` 같은 상대 시간)
- 항목 개수 카운트 (예: "지금까지 12명이 등록")
- 같은 선수를 좋아하는 사람끼리 색깔 매칭 (예: 같은 이름은 같은 색 배지)
- 입력 후 폼 자동 비우기 + 이름 input에 포커스
- 모바일 화면 대응

> 시간 남으면, 안 되면 패스 OK.

## 6. 안 만들어도 되는 것 (Out of scope)

- 로그인/사용자 인증
- 항목 삭제/수정 (이번엔 등록과 조회만)
- 댓글, 좋아요 등 소셜 기능
- 검색/정렬 UI (최신순 고정이면 충분)
- 관리자 페이지

## 8. 디자인 가이드 (예쁘게!)

- **테마**: 잔디(축구장) 분위기 살짝 — 메인 컬러 **그린 계열** + 화이트/연한 그레이
- **포인트 컬러**: 등록 버튼은 진한 그린, 호버 시 살짝 어두워짐
- **타이포**: Pretendard / Noto Sans KR 등 한글이 예쁜 폰트, 제목은 굵게
- **레이아웃**: 가운데 정렬 한 컬럼, 최대 폭 560~640px
- **카드 디자인**: 둥근 모서리(8~12px), 미세한 그림자, 호버 시 살짝 떠오르기
- **입력 필드**: 큼직하게 (높이 44px+), 포커스 시 그린 외곽선
- **상대 시간**: 옅은 회색
- **이모지**: ⚽ 📋 같이 살짝만 (과하지 않게)
- **여백 넉넉**: 답답하지 않게

> 100% 강제는 아니고, 랑이 감각 더 좋으면 그쪽으로! 다만 "예쁘게"는 꼭 신경 써주세요.

## 9. 기술 권고 (참고용)

> ⚠️ 핵심 제약: 데이터가 **외부에 저장**되어야 해요. 이유 두 가지 —
> ① 다른 기기/사용자가 같은 목록을 봐야 함, ② 모가 입력 이벤트를 외부에서 관찰해서 알림을 보낼 수 있어야 함.
>
> → **`localStorage` 단독은 안 됩니다.**

권장 옵션 (랑이가 편한 거 고르세요):

| 방식 | 특징 |
|---|---|
| **Firebase Realtime DB / Firestore** | 무료 티어 충분, 실시간 구독 가능, 모가 webhook 또는 polling으로 모니터링 가능 |
| **Supabase** | Postgres 기반, 무료, REST/Realtime 둘 다 가능 |
| **jsonbin.io / Pantry / Replit DB** | 가벼운 KV 저장소, 빠르게 붙이기 좋음 |
| **자체 작은 API + Vercel/Render 무료 호스팅** | 완전 커스텀이 좋다면 |

배포:
- **GitHub Pages** 추천 (사용자 요청). 정적 호스팅이라 백엔드는 위 BaaS 중 하나로 분리.
- 또는 Vercel/Netlify 도 OK — 사용자가 GitHub Pages를 예시로 들었지만 동급이면 무방.

기존 repo 구조와 어울리게:
- `index.html` 한 페이지 정적 사이트가 이미 있으니, 거기에 추가하거나 별도 `players.html`로 빼고 link 거는 방식 둘 다 OK.
- 키/시크릿은 클라이언트 노출되니 **읽기/쓰기 권한이 좁게 제한된 키**만 쓰세요. (Firebase 룰, Supabase RLS 등)

## 10. 모니터링 인터페이스 (모와 협업)

모가 **새 항목 등록을 외부에서 감지**할 수 있어야 합니다. 다음 중 하나는 보장해주세요:

- 선택 A: 저장소가 **공개 read API**를 제공 → 모가 polling 가능
- 선택 B: 저장 시 **webhook** 발사 (Firebase Functions, Supabase Edge Functions, Zapier 등) → 모 채널/슬랙으로 바로 알림 가능
- 선택 C: 저장소 접속 정보 (URL/anon key/콜렉션명)를 모에게 공유 → 모가 직접 SDK 붙여서 구독

랑이가 어떻게 만들었는지에 따라 모와 짧게 맞추면 돼요. 완료 후 모 멘션할 때 **저장소 종류 + 읽는 방법**을 같이 알려주세요.

## 11. 받아들이는 기준 (Acceptance Criteria)

다음이 다 되면 "완성"이에요.

1. **상용 링크가 발급되어** 누구나 접속 가능 (예: `https://<gh-username>.github.io/<repo>/players.html`)
2. 페이지에서 이름/선수 입력 후 등록 → **즉시 하단 목록에 표시**
3. **새로고침/다른 브라우저**에서 열어도 동일한 목록이 보임
4. 빈 입력으로 등록 시도 시 안내 문구 표시, 등록 안 됨
5. 디자인이 "기획서 8번"의 그린 톤 가이드를 따름 (대충이라도 OK, 너무 못생기지만 않으면 ✅)
6. 모가 **외부에서 입력 이벤트를 감지할 방법**이 존재 (10번 항목 중 하나)

## 12. 다음 단계

1. **랑이**: `rang` 브랜치에서 구현 → 상용 링크 배포 → 완료 시 스레드에 **모 멘션** + 링크/저장소 정보 공유
2. **모**: 랑이 완료 알림 받으면, 입력 이벤트 모니터링/알림 환경 구축 검토. 간단하면 직접 구축, 복잡하면 사용자에게 보고.
3. **사용자**: 모니터링까지 붙으면 직접 써보고 피드백.

질문이나 막히는 부분 있으면 스레드에서 솜이 멘션해주세요. 🐰

## 변경 이력 — v2 (2026-05-10): 선수 사진 추가

### 무엇이 바뀌나
입력 폼에 **선수 사진** 필드가 하나 더 들어가요. 등록한 카드에도 사진 썸네일이 같이 보입니다.

### 어떤 사진을 받나 (가정)
**좋아하는 축구선수의 사진**을 기본으로 가정합니다 (예: 손흥민 얼굴 사진). 본인 사진/셀카가 의도라면 알려주세요 — 카드 레이아웃이 달라져요.

### 추가 Must
| 항목 | 설명 |
|---|---|
| 파일 업로드 input | 갤러리에서 사진 선택. 모바일은 카메라로도 찍을 수 있게 (`accept="image/*"`, `capture` 속성) |
| 미리보기 | 등록 전 폼에서 사진을 작게 보여줌 |
| 외부 호스팅 | 이미지 호스팅 서비스에 업로드 후 **URL만** JSONBin에 같이 저장 |
| 카드 썸네일 | 카드 좌측에 정사각/원형 썸네일 (60~80px) |
| 사진 없는 항목 처리 | placeholder 이미지 또는 이니셜/⚽ 배지 |

### 사진 필수? 선택?
**선택 (Optional)** 으로 잡았어요. 신규 사용자가 사진 없이도 빠르게 등록 가능해야 함. 필수로 원하시면 알려주세요.

### 검증
- 형식: `jpg` / `png` / `webp`
- 크기: 5MB 이하 (초과 시 친절한 안내, 등록 막힘)
- 클라이언트 사이드 리사이징 권장 (max width 800px, JPEG quality 0.8) — 업로드 속도/호스팅 용량 둘 다 절약

### 저장 옵션 (랑이 선택)
| 방식 | 셋업 | 메모 |
|---|---|---|
| **imgbb** | 회원가입 → API key 1개 | 가장 간단. 32MB까지, 영구 호스팅. URL만 JSONBin에 저장 |
| **Cloudinary** | cloud name + unsigned upload preset | CDN/자동 리사이징. 기능 많지만 셋업 살짝 더 |
| **base64 → JSONBin 직접 저장** | 셋업 0 | 누적되면 bin 폭발. 비추천 |
| **GitHub repo commit** | write token 클라이언트 노출 | 보안상 비추천 |

→ **imgbb 추천**. 사용자가 키 발급해주면 바로 진행.

### 디자인 가이드 추가
- 카드 썸네일: 60~80px 정사각, 둥근 모서리 (50% radius로 원형도 OK)
- 빈 슬롯 (사진 없는 카드): 옅은 그레이 배경 + 이니셜 또는 ⚽ 아이콘
- 폼 안 미리보기: 입력 영역 우측/하단에 100~120px 정사각, 사진 변경/삭제 버튼

### 데이터 모델 변경
- 추가 필드: `photoUrl?: string` (optional)
- 기존 등록 항목: `photoUrl` 없음 → placeholder로 표시

### 모니터링 영향 (모 참고)
- 신규 항목 알림 메시지에 `photoUrl` 같이 포함 → 슬랙 자동 unfurl로 썸네일 보임
- `bot/src/monitor.ts` 메시지 빌더에 `photoUrl` 분기 1줄 추가하면 됨

### 수락 기준 추가
7. 사진을 첨부해 등록 → 카드에 썸네일이 보임
8. 사진 없이 등록 → placeholder 표시, 동작 정상
9. 5MB 초과 사진 업로드 시 친절한 안내, 등록 막힘
10. 신규 알림 슬랙 메시지에 사진 미리보기가 같이 보임 (URL 첨부 시 자동 unfurl)

### v2 다음 단계

## 변경 이력 — v2.1 (2026-05-10): MVP 스코프 축소

태진님 피드백("40분이면 길다, 썸네일 빼도 OK") 반영. v2 Must에서 작업량 큰 항목들을 Out of scope로 옮겼어요.

### 빠진 것 (Out of scope)
| 항목 | 사유 |
|---|---|
| 폼 내 미리보기 슬롯 (108×108) | 시간 큼, 핵심 기능 아님 |
| 클라이언트 사이드 리사이징 (canvas, max 800px) | imgbb가 32MB까지 받음, MVP는 원본 업로드 OK |
| 카드 원형 썸네일 (64px) | 사용자 명시적 OK |
| 이니셜 fallback placeholder | 썸네일 자체 빠지니 불필요 |

### 유지되는 Must (단순화)
| 항목 | 설명 |
|---|---|
| 사진 input | `<input type="file" accept="image/*">` 한 줄, optional |
| imgbb 업로드 | 등록 버튼 누르면 imgbb POST → URL 받기 (5MB 한도만 체크) |
| 데이터 저장 | `entry.photoUrl?: string` JSONBin에 같이 PUT |
| 카드 표시 | `photoUrl` 있으면 평범한 `<img src>` (자연 크기, CSS 디테일 없음). 없으면 안 보여줌 |
| graceful degrade | imgbb 키 없으면 사진 영역 숨김, 텍스트 등록은 정상 |

### 알림 영향 (모 변경 없음)
- 알림 메시지에 `photoUrl` URL 한 줄 추가 → 슬랙 자동 unfurl로 사진 보임
- 모가 정리한 라우팅/자가복구 골격은 그대로 유효

### 사용자 결정 사항
- **imgbb API key 발급 완료**: 스레드 공유됨
- 본인 셀카 vs 선수 사진 가정은 그대로 (선수 사진)

### 예상 시간
랑이 1차 작업분에서 위 4개 빼면 **10~15분** 추가로 마무리 가능.

## v2.1 다이어트 — 2026-05-10 (사용자 요청: 빨리 끝내자)

40~60분 예상이 길다는 피드백 → **15분 내** 끝낼 수 있게 비핵심 항목 제거.

### 빼는 것
- **클라이언트 사이드 리사이징** (canvas/quality 조정) — imgbb가 32MB까지 받으니 원본 그대로 업로드
- **미리보기 슬롯** — 폼에서 사진 미리보기 안 보여줘도 OK
- **카드 썸네일 처리** (원형/이니셜 placeholder) — 사용자 직접 "썸네일이 문제면 빼도 돼". 카드엔 사진 있으면 그냥 `<img>` 태그 (CSS `max-width: 100%`), 없으면 아예 표시 안 함
- **형식·크기 검증** — imgbb가 거부하면 그 에러만 사용자에게 그대로 전달
- **graceful degrade (키 미설정 분기)** — 키 이미 있음, 분기 불필요

### 남기는 것 (Minimal Must)
| 항목 | 설명 |
|---|---|
| 사진 input | `<input type="file" accept="image/*">` 한 줄 |
| imgbb 업로드 | 등록 시 imgbb POST → URL 받기 |
| JSONBin 저장 | `photoUrl` 필드만 같이 저장 (사진 없으면 필드 생략) |
| 카드 표시 | `photoUrl` 있으면 `<img src>` 1개. 없으면 표시 안 함 |

### 데이터 모델 (변경 없음)
```json
{ "id": "...", "name": "...", "player": "...", "photoUrl": "https://i.ibb.co/...", "createdAt": 1746... }
```

### imgbb 호출 (참고)
```
POST https://api.imgbb.com/1/upload?key=API_KEY
form-data: image=<base64 또는 file>
→ data.url 사용
```

### 수락 기준 (다이어트)
1. 사진 첨부 → 등록 → 카드에 사진 보임
2. 사진 없이 등록 → 카드에서 사진 영역 없음, 동작 정상
3. imgbb 업로드 실패 시 사용자에게 에러 그대로 노출 (앱 안 멈춤)

→ 디자인/리사이징/썸네일은 v2.2에서 추후 보강 가능. 지금은 동작이 우선.
